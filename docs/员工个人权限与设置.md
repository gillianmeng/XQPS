## 一、角色定义与来源

### 1.1 飞书档案字段「角色」

| 字段 | 说明 |
|------|------|
| **角色** | 飞书多维表字段，用于区分用户类型 |

**可能取值：**
- `员工`：普通员工，有下属或参与团队场景
- `管理者`：有下属的管理者
- `员工个人`：纯个人贡献者，无团队相关权限

### 1.2 应用内角色映射

```python
# 代码位置：new_app.py ~1071-1073
role_from_record = extract_text(fields.get("角色"), "").strip()
if st.session_state.role not in ["员工", "管理者"]:
    st.session_state.role = role_from_record if role_from_record in ["员工", "管理者"] else "员工"
```

- `st.session_state.role` 仅取 `"员工"` 或 `"管理者"`
- 当飞书「角色」为 `员工个人` 时，`role_from_record` 不在 `["员工","管理者"]`，会 fallback 为 `"员工"`
- 因此：**员工个人** 在应用内仍按 `role = "员工"` 处理，仅通过 `role_from_record` 控制部分展示

## 三、员工个人可见能力

### 3.1 员工自评模块（📝 员工自评）

| 能力 | 说明 |
|------|------|
| 工作模块 | 工作目标 1–5 及总结、权重、自评得分 |
| 能力模块 | 通用能力总结、自评得分（无领导力，因 role=员工） |
| 保存草稿 | 可暂存至飞书，不锁定 |
| 确认提交 | 提交后 `自评是否提交=是`，表单锁定不可再改 |

**权重规则：**
- 员工：工作模块 80%，通用能力 20%
- 管理者：工作 60%，通用能力 20%，领导力 20%

### 3.2 自评提交后

- 所有输入框、下拉框、按钮置为 `disabled=is_submitted`
- 显示「🔒 您的自评已提交，当前表单不可修改。」

## 四、员工个人不可见 / 不可用

| 模块 | 说明 |
|------|------|
| 👥 上级评分 | 仅 `role == "管理者"` 可见 |
| 📌 一级部门负责人调整 | 仅 `role == "管理者"` 且 `is_dept_head` 可见 |
| 📌 分管高管调整 | 仅 `role == "管理者"` 且 `is_vp` 可见 |
| 📊 视图与报表 | 仅 `role == "管理者"` 可见 |
| 📂 团队历史绩效 | `role_from_record == "员工个人"` 时整个 Tab 不显示 |

## 五、侧边栏展示（员工个人）

员工个人与普通员工侧边栏一致：

- 欢迎语
- 绩效考核周期
- 直接评价人、一级部门负责人、分管高管、HRBP
- 员工信息：姓名、工号、部门、岗位、角色
- 退出登录

**无：**
- 验证模块（仅管理者评估下属时显示）
- 下属选择器

## 六、数据与校验

### 6.1 员工自评校验（step1_can_submit）

| 校验项 | 规则 |
|--------|------|
| 权重 | 工作目标权重之和 = target_weight（员工 80%） |
| 工作目标 | 每项有总结（100–5000 字）、有自评得分 |
| 通用能力 | 有总结（100–5000 字）、有自评得分 |
| 领导力 | 仅管理者校验 |

### 6.2 飞书写入字段（自评提交）

- 工作目标 1–5 及总结、权重、自评得分
- 通用能力总结、通用能力自评得分
- 自评得分、自评等级
- 自评是否提交 = 是

## 七、演示登录

`demo_users.json`、`demo_users_hr.json`、`demo_users_wealth.json` 中 `role` 仅支持 `"员工"` 或 `"管理者"`，无 `"员工个人"`。

默认合并：`?demo_entry=1` 或 `?demo_entry=1&demo_dept=all` 时，会合并三部门（人力资源部、研发质量保障部、财富顾问部）所有人员。单部门：`?demo_dept=hr`、`?demo_dept=wealth`、`?demo_dept=rd`。

员工个人需在飞书多维表「角色」字段中配置为 `员工个人`，正式登录后由飞书档案决定。
